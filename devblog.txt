  v0.6c — Kill Economy Rework + Speed Fix               2026-03-04
--------------------------------------------------------------------------------

Knock and elimination are now two separate stats with distinct reward timing.
Equipment speed bonuses finally drive actual movement. Wisps can die in the
abyss. Storm no longer auto-starts so you can actually test things in peace.

WHAT CHANGED:

  [GAS → CMC Speed Wiring — BUG FIX]
  - Equipment boots modify GAS MoveSpeed/GlideSpeed attributes, but the
    CharacterMovementComponent was returning hardcoded base values
  - Added GetGASMoveSpeed() and GetGASGlideSpeed() helpers to CMC
  - GetMaxSpeed() now reads from GAS attributes at runtime (with fallback)
  - PhysGliding() uses GAS GlideSpeed for target velocity + acceleration
  - Result: DebugEquipTestBoots now actually makes you walk/glide faster

  [Knock vs Elimination — GAMEPLAY CHANGE]
  - NEW: "Knock" = sending a hunter to wisp form. Separate stat from kills.
  - NEW: "Elimination" = wisp fully dies (expired, executed, or abyss death)
  - Knocking a player no longer awards ANY rewards (no gold, no shards, no
    kill credit). Rewards are deferred to the elimination event.
  - Added Knocks replicated property to PlayerState (separate from Kills)
  - Knocker tracked on the wisp via KnockerPlayerState reference

  [Kill Reward Timing — GAMEPLAY CHANGE]
  - OLD: killer got gold transfer + shard drops + kill count on knock
  - NEW: rewards only on elimination:
    * Wisp HP expires → knocker gets kill credit + shard drops at wisp loc
    * Wisp executed (E key) → executor gets kill credit + heal + shard drops
    * Abyss death → killer gets knock + kill credit + shards near killer
  - Gold is NO LONGER transferred on kill (stays with victim for deathbox)
  - Shards always spawn as world pickups at the death location

  [Abyss Death Edge Cases — NEW]
  - Direct abyss kill (DebugTriggerSpike over void): counts as BOTH knock
    and elimination simultaneously (no wisp phase)
  - Shard pickups spawn near the KILLER's last grounded position (not void)
  - Wisp walks off map edge: wisp NOW DIES (was previously immune to abyss)
  - Wisp abyss death: shards spawn at victim's pre-wisp grounded location
  - Added ServerKillFromAbyss() to BBWispPawn
  - AbyssKillVolume updated to kill wisps instead of ignoring them

  [Storm Manual Start — CHANGE]
  - Storm bAutoStart changed from true to false
  - Added DebugStartStorm console command to start storm on demand
  - No more racing against storm damage while trying to test equipment

  [Deprecated API Migration — CLEANUP]
  - Removed InheritableOwnedTagsContainer usage (deprecated in UE 5.3+)
  - Storm damage and consumable heal GEs now use Spec.DynamicGrantedTags
  - Fixed in both BBStormManager.cpp and BBInventoryManager.cpp

  [Compile Error Fixes]
  - Added missing ABreachbornePlayerState forward declaration in GameMode.h
    (was root cause of 5 cascading errors: C2061, C2660, C2511, C2352)

  v0.6b — Debug HUD + Wisp Rez Rework                  2026-03-04
--------------------------------------------------------------------------------

Visual feedback for every gameplay system — health, wisp state, inventory,
storm, glider heat — all drawn via Canvas with zero UMG dependency. Plus a
critical rework to wisp proximity rules so rezzing actually feels right.

WHAT'S NEW:

  [Debug Overlay — "DebugHUD" console command]
  - Semi-transparent panel in the top-left corner
  - Player info: name, team, life state, level, XP, kills
  - Health: HP/MaxHP, Shield/MaxShield, Armor
  - Combat: ATK, AP, CDR%, Crit% (x multiplier)
  - Movement: MoveSpeed, GlideSpeed, DashDistance
  - Glider: state + heat bar with blue-to-red color gradient (hunter only)
  - Wisp: state, wisp HP%, rez bar% (wisp form only)
  - Inventory: gold, shards, weapon/helmet/boots with tier, armor tier + HP,
    powers, full backpack contents
  - Storm: distance to storm center
  - All data read from replicated client-side state — no RPCs

  [Gameplay HUD — "ToggleHUD" console command, ON by default]
  - Bottom-center health bar with color shift: green > 60%, yellow > 30%, red
  - Shield bar (cyan) below health, only shown if MaxShield > 0
  - Armor bar colored by tier: white/green/blue/purple
  - Team + Level label left of bars, Gold + Shard count on the right
  - When in wisp form: health bars replaced by wisp HP bar (orange) + rez
    bar (green) with percentage readout
  - Numeric values displayed beside each bar

  [3D Debug Shapes — "DebugShapes" console command]
  - Green circle (400cm) at each wisp: proximity rez/stomp radius
  - Red circle (150cm) at each wisp: execute interaction radius
  - Line from player pawn to storm center (green if close, red if far)

  [Wisp Proximity Rework — GAMEPLAY CHANGE]
  - OLD: HP always drained every tick. Rez bar filled whenever ally was
    present, even with enemy contesting. Both happened simultaneously.
  - NEW priority rules:
    * Ally only:        HP decay STOPS, rez bar fills
    * Both ally+enemy:  Enemy takes priority — HP drains at stomp rate,
                        rez bar STOPS filling and decays
    * Enemy only:       HP drains at stomp rate, rez bar decays
    * Nobody:           HP drains at base rate, rez bar decays
  - Uncontested ally rez is now safe and steady
  - Enemy arrival completely shuts down the rez and forces a fight

  - UI/BBDebugHUD.cpp           (Canvas drawing: overlay, bars, 3D shapes)

  - Core/BreachborneGameMode.cpp   (HUDClass = ABBDebugHUD, include added)
  - Wisp/BBWispPawn.h              (2 public getters: GetProximityRadius,
                                    GetExecuteRadius)
  - Wisp/BBWispPawn.cpp            (ServerWispTick rewritten: ally-only pauses

  v0.6.0 — Phase 5: Item System & Match Economy        2026-02-19
--------------------------------------------------------------------------------

The full Open Beta-style item system is in. Equipment slots with upgrade
evolutions, powers that grant abilities, consumables, armor in the damage
pipeline, gold/shard economy, world pickups, shopkeepers, and PvP kill
loot drops. All server-authoritative, all replicated. Testing is via debug
console commands — no UI yet.

WHAT'S PLAYABLE:

  [Equipment System — Weapon, Helmet, Boots]
  - 3 equipment slots: Weapon, Helmet, Boots
  - Equipping applies Infinite-duration programmatic GEs to the ASC
    (same NewObject<UGameplayEffect> pattern as StormManager)
  - Equipment modifies all attributes: AttackPower, AbilityPower, MoveSpeed,
    GlideSpeed, DashDistance, MaxHealth, MaxShield, CooldownReduction,
    CritChance, CritMultiplier
  - Unequipping removes the GE cleanly — attributes revert
  - GE handles tracked per-slot on PlayerState (server-only, non-replicated)

  [Upgrade & Evolution]
  - Equipment starts at tier 0, upgradeable with Upgrade Shards
  - Stats scale with tier: base * (1.0 + tier * 0.5)
  - At tier 2+, choose 1 of 3 evolution paths (A, B, C)
  - Evolution adds bonus stats on top of tier scaling
  - Evolution choice is permanent for the match
  - Old GE removed and new GE applied on every upgrade/evolve

  [Armor — Damage Pipeline Integration]
  - Armor attribute added to BBHealthSet (was a Phase 5 placeholder comment)
  - BBDamageExecution captures target Armor, subtracts flat reduction
  - Minimum 1 damage floor (armor can't fully negate hits)
  - 4 armor tiers: White (5), Green (10), Blue (20), Purple (35)
  - SetArmorTier sets both the Armor attribute value and armor HP/durability
  - Armor replicates through the existing FRepHealthData proxy struct

  [Powers — 2 Slots]
  - Powers grant a GameplayAbility via ASC->GiveAbility on equip
  - Powers also apply stat bonus GEs (same equipment GE pattern)
  - Unequip clears the ability and removes the stat GE
  - Passive powers auto-activate; activatable powers bind to input tags
  - Ability spec handles tracked per-slot on PlayerState

  [Consumables — Backpack]
  - 4-slot backpack (TArray<FConsumableStack>)
  - Stacking: same consumable type stacks up to MaxStackSize
  - Food heals instantly (applies IncomingHealing meta attribute)
  - ArmorShards repair +50 armor HP
  - UseConsumable decrements count, removes empty stacks
  - Hotkeys 1-4 mapped to backpack slots

  [Gold & Shard Economy]
  - Gold: earned from PvE/PvP, spent at shopkeepers
  - Upgrade Shards: collected from kills and objectives, spent on upgrades
  - PvP kill: killer gets 50% of victim's gold (server transfers automatically)
  - PvP kill: 3 shard pickups spawn at death location

  [World Item Pickups]
  - ABBWorldItem: replicated pickup actor with sphere collision
  - Supports equipment, gold, shards, consumables, and powers
  - E key interaction: traces for nearest world item, server validates
  - Items hide on pickup for all clients via OnRep_IsPickedUp
  - FBBWorldItemSpawner: static helpers to spawn any pickup type

  [Shopkeeper]
  - ABBShopkeeper: fixed-location actor with configurable shop listings
  - Server validates: proximity, gold balance, listing validity
  - Purchase deducts gold and equips item in one transaction

  [Item Registry]
  - UBBItemRegistry: GameInstanceSubsystem (persists across level loads)
  - Scans UAssetManager for all BBEquipment, BBPower, BBConsumable assets
  - Lookup by ID, random drop generation for future PvE loot tables
  - Manual registration methods for testing

  [Data Asset Architecture]
  - UBBEquipmentDefinition: ItemID, SlotType, BaseStats, 3 evolutions,
    shard costs per tier. UPrimaryDataAsset with asset type "BBEquipment"
  - UBBPowerDefinition: PowerID, GrantedAbilityClass, bIsPassive,
    StatBonuses, bDropsOnDeath. Asset type "BBPower"
  - UBBConsumableDefinition: ConsumableID, ConsumableType, MaxStackSize,
    heal/repair amounts, OnUseEffectClass. Asset type "BBConsumable"

  [Debug Commands (all via console)]
  - DebugAddGold <N>          — add gold to inventory
  - DebugAddShards <N>        — add upgrade shards
  - DebugEquipTestWeapon      — equip a test weapon (+15 AP, +10 AbP)
  - DebugEquipTestHelmet      — equip a test helmet (+50 MaxHP, +25 MaxShield)
  - DebugEquipTestBoots       — equip test boots (+100 MoveSpeed, +50 GlideSpeed)
  - DebugUnequip <Slot>       — unequip from Weapon/Helmet/Boots
  - DebugUpgrade <Slot>       — upgrade equipment tier (costs 5 shards)
  - DebugEvolve <Slot> <Path> — evolve equipment (A/B/C)

  - DebugAddConsumable <Name> <Count> — add Food/ArmorShard consumables

  v0.5.0 — Phase 4: Storm System                       2026-02-14
--------------------------------------------------------------------------------

The storm is in. A shrinking safe zone now forces squads toward each other —
the fundamental BR pacing mechanic. Hunters outside the green circle take
escalating DOT damage that flows through the existing GAS pipeline, meaning
storm damage automatically triggers spiking on gliders with zero extra code.

WHAT'S PLAYABLE:

  [Storm — Shrinking Safe Zone]
  - 4-phase storm with escalating damage (5 -> 10 -> 20 -> 40 per tick)
  - Circle shrinks toward map center over time, then holds, then shrinks again
  - Final phase collapses to zero radius — nowhere is safe
  - Damage ticks faster in later phases (1.0s -> 0.5s interval)
  - Storm damage flows through BBDamageExecution — shield absorbs first,
    then health, exactly like combat damage
  - Gliding outside the storm = spiked (storm damage triggers the spike
    check in BBHealthSet automatically)
  - Dead characters are not damaged (State_Dead skip)
  - All characters with IAbilitySystemInterface take storm damage
    (hunters, GlideBots, target dummies — anything damageable)

  [Debug Visualization]
  - Green debug circle = current safe zone boundary
  - Yellow debug circle = target zone (where the circle is shrinking toward)
  - Circles drawn on both server and all clients via replicated state
  - Output log banners: "=== STORM PHASE 2 ===" etc.

  [Storm Shift Architecture]
  - Pluggable storm shift mutators via UStormShiftBase UObject
  - Default shift: 4 phases, shrinks to center, linear damage escalation
  - Adding new shifts (Nomadic, BulletTrains, GlobalOracle) = subclass
    UStormShiftBase, override GetPhaseConfigs/PickNextCenter/ModifyDamage
  - StormShiftClass configurable on the StormManager in editor

  [Architecture]
  - ABBStormManager: replicated actor with own ASC (actor-owned, like
    TargetDummy/GlideBot pattern)
  - Programmatic UGameplayEffect with BBDamageExecution — no Blueprint
    assets needed
  - Spawned automatically by GameMode in InitGameState
  - bAlwaysRelevant so all clients receive storm circle state

FILES ADDED (5):
  - Storm/StormTypes.h              (EStormPhaseState, FStormPhaseConfig)
  - Storm/StormShiftBase.h          (abstract base for shift mutators)
  - Storm/StormShift_Default.h/.cpp (4-phase default shift)
  - Storm/BBStormManager.h/.cpp     (main storm actor + damage + debug draw)

FILES MODIFIED (4):
  - Abilities/BBGameplayTags.h/.cpp   (Effect_StormDamage tag)
  - Core/BreachborneGameMode.h/.cpp   (spawn StormManager in InitGameState)

KNOWN ISSUES / TODO:
  - No storm VFX (just debug circles — visual wall/fog deferred to art pass)
  - No storm UI indicator (minimap circle, damage warning)
  - No storm shift mutators beyond Default (Nomadic, BulletTrains planned)
  - Storm doesn't pause during drop phase (needs match phase integration)
  - No day/night cycle interaction yet

WHAT'S NEXT:
  - Phase 5: Item system — equipment slots, upgrade shards, shopkeepers,
    powers, consumables, gold economy
  - Or: Additional storm shifts (Nomadic, BulletTrains, GlobalOracle)
  - Or: Wisp system (Phase 6) for the downed-state revival chain

  v0.4.0 — Ghost Combat + Spiking Fully Playable       2026-02-13
--------------------------------------------------------------------------------

Ghost is a real character now. All five abilities deal damage, projectiles fly,
grenades explode, napalm burns. Spiking — the signature "hit them while they're
gliding and they plummet" mechanic — is fully testable end-to-end for the first
time. Shoot a gliding target, watch them spike, watch them fall into the abyss.

WHAT'S PLAYABLE:

  [Ghost — Full Combat Kit]
  - LMB: Auto-rifle. Single projectile per click, 25 damage, 3500 speed.
    Projectiles replicate, server validates hits.
  - RMB: Spike Grenade. Arcs toward cursor, detonates on enemy contact or
    after 3s fuse. 400-unit AoE, 80 damage to all enemies in radius.
  - Q: Piercing Laser. Instant line trace in cursor direction, pierces through
    all targets in the line. Damage via GAS pipeline.
  - R: Napalm Strike. Drops a 500-unit radius fire zone at cursor location.
    Ticks 40 damage every 0.5s for 5 seconds. Server-spawned, DOT via GAS.
  - Passive: Kill resets Shift cooldown. Wired through OnHealthDepleted.
  - Shift: Combat Slide (unchanged from v0.3.0).

  [Spiking — Now Testable]
  - Shoot a gliding hunter or bot = instant SPIKE (was untestable in v0.3.0)
  - Spiked target stunned, fast-falls straight down for 1.5 seconds
  - Spike over the abyss = instant kill with kill credit to the spiker
  - AbyssKillVolume fully functional with spike vs. fall-off distinction
  - Output log banners: "=== SPIKED ===" / "=== ABYSS SPIKE KILL ==="

  [Damage Pipeline]
  - BBDamageExecution calculates final damage (armor/shield absorption)
  - Shield absorbs first, overflow goes to health
  - Health depletion fires OnHealthDepleted delegate (feeds kill tracking,
    respawn, future wisp system)
  - SetByCaller_Damage tag for flexible per-ability damage values
  - All damage server-authoritative, replicated via GAS attribute system

  [Test Actors]
  - BBTargetDummy: stationary damage sponge. Place in level, shoot it,
    watch the full damage pipeline in the output log. Auto-resets health
    after 3 seconds. Team 99 (always targetable by players).
  - BBGlideBot: gliding patrol target. Flies between two editor-draggable
    waypoints with glider open. Shoot it to test spiking. Place over ground
    for spike-then-recover, place over the abyss for spike-kill.
    Auto-respawns after 3 seconds. Configurable patrol speed and team ID.

  [Architecture Improvements]
  - GliderComponent decoupled from AHunterCharacter — now works on any
    ACharacter that implements IAbilitySystemInterface
  - ForceOpenGlider/ForceCloseGlider for server-side bot/AI control
    (bypasses RPCs that require an owning client connection)
  - AbyssKillVolume uses IAbilitySystemInterface + FindComponentByClass
    instead of hardcoded AHunterCharacter casts
  - BBHealthSet spike detection uses FindComponentByClass<UGliderComponent>
  - All damage sources (projectile, grenade, napalm) recognize
    HunterCharacter, TargetDummy, and GlideBot as valid targets

  v0.3.0 — Phase 3: Gliding & Spiking System          2026-02-09
--------------------------------------------------------------------------------

The core aerial movement system is in. Hunters can now glide across the map
and dash through the air. The floating-island fantasy is starting to come
together.

WHAT'S PLAYABLE:

  [Gliding]
  - Hold Space while airborne to open the glider
  - Glider always flies toward the mouse cursor (not WASD)
  - Gradual linear acceleration: takes ~1 second to reach full glide speed
  - Sharp turns (>90 degrees) cause a braking effect — you slow down before
    picking up speed in the new direction. Feels like real inertia.
  - Reduced gravity while gliding (15% of normal) — floaty and smooth
  - Heat builds while gliding. Hit 100% heat = auto-close (overheat)
  - Heat decays when grounded
  - Minimum altitude check prevents opening glider too close to the ground
  - Landing auto-closes the glider and resets heat
  - Abilities blocked while gliding (must close glider to use them)
  - Glider indicator (flat plane) appears above character while gliding

  [Dash (Ghost Shift)]
  - Press Shift to dash in the cursor direction
  - Works on ground and in air
  - Dash then immediately glide = momentum carries over (speed decays
    gradually from dash speed down to normal glide speed)
  - Instant velocity burst (2000 units) with slight upward lift
  - 6 second cooldown via GAS
  - Blocked while gliding, spiked, or stunned

  [Visual Indicators]
  - Cone mesh on each character shows facing direction (cursor aim)
  - Flat plane hovers above the character while gliding — visible to all players
  - Glider indicator appears/disappears on state change for all clients

CODE COMPLETE BUT NOT YET TESTABLE:

  [Spiking] (needs damage-dealing abilities to trigger)
  - Taking ANY damage while gliding = SPIKED
  - Spiked = stunned, no input, fast-fall straight down
  - All abilities blocked while spiked (State_Spiked tag)
  - Spike lasts 1.5 seconds, then auto-recovers
  - Damage hook wired in BBHealthSet — will activate once Ghost's combat
    abilities (LMB/Q) deal real damage

  [Abyss Kill Volume] (needs to be placed in editor)
  - AbyssKillVolume actor class created — kills on overlap, server-only
  - Must be manually placed below the map in the level editor to function
  - Deathbox spawning stubbed until Phase 6 (Wisp system)

  - GliderComponent: server-authoritative replicated state machine

  - Server RPCs for open/close requests, server validates everything
  - Movement mode replication handled automatically by CMC
