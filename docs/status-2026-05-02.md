# Coding Standards Audit

## Files that follow standards correctly

- `BBAbilitySystemComponent.h/.cpp` — `BB` prefix, `UPROPERTY`/`UFUNCTION` where needed, `Mixed` replication mode set in constructor, `SetIsReplicatedByDefault(true)`. The reflective removal of `SpawnedAttributes` from the rep list is novel but documented.
- `BBHealthSet.h/.cpp` — Correctly uses `ATTRIBUTE_ACCESSORS`, `IncomingDamage` is a meta attribute, `PostGameplayEffectExecute` follows GAS canonical pattern, blocks damage on wisp/invulnerable tags.
- `BBDamageExecution.h/.cpp` — Standard `FBBDamageStatics` pattern; clean attribute capture; `Source` vs `Target` capture sides correct.
- `BBGameplayAbility.h/.cpp` — Adds blocked-state tags in constructor; clean accessors; `InstancedPerActor`.
- `BBGameplayTags.h/.cpp` — `UE_DECLARE_GAMEPLAY_TAG_EXTERN` / `UE_DEFINE_GAMEPLAY_TAG` pattern (compile-safe, no string typos).
- `BBStormManager.h/.cpp` — Server-only ticking, `HasAuthority()` guards, replicated state for client visualization.
- `BBWispPawn.h/.cpp` — Server-tick proximity logic, replicated state, delegates fired only on server, executor damage delegate cleanup on cancel.
- `BBDebugHUD.h/.cpp` — Correctly subclasses `AHUD` (which UE only creates on clients); uses Canvas; safe for dedicated server.
- `Breachborne.Build.cs` / `*.Target.cs` — `BuildSettingsVersion.V6`, `EngineIncludeOrderVersion.Unreal5_7`, `Server` target type defined.
- `Hunters/Ghost/GA_Ghost_*.h/.cpp` — All set `NetExecutionPolicy` correctly per the canonical table (LMB/RMB/Q/Shift = LocalPredicted, R = ServerInitiated, Passive = ServerOnly). All bind input via `AbilityInputTag = BBGameplayTags::InputTag_*`.

## Networking-mistake patterns I looked for

I cross-checked each item in CLAUDE.md's "Common Mistakes" list:

1. **Forgetting `DOREPLIFETIME`** — All replicated UPROPERTYs found via grep have a corresponding `DOREPLIFETIME` call. ✅ clean.
2. **Forgetting `Super::GetLifetimeReplicatedProps`** — Every override I read calls it first. ✅ clean. (`BBAbilitySystemComponent::GetLifetimeReplicatedProps` calls Super and *then* mutates the list to remove `SpawnedAttributes` — intentional; documented in comments.)
3. **`GetGameMode()` on clients** — Zero call sites in source. The only mention is the comment in `BreachborneGameMode.h` warning against it. ✅ clean.
4. **UMG widgets in server code paths** — Zero `CreateWidget` / `NewObject<UUserWidget>` in source. `BBDebugHUD` uses Canvas/`AHUD` only, which UE creates on clients only. ✅ clean.
5. **Missing `_Implementation` suffix** — All `UFUNCTION(Server, ...)` / `UFUNCTION(Client, ...)` declarations I sampled have matching `_Implementation` definitions. ✅ clean.
6. **Raw `ApplyDamage` / `TakeDamage`** — Zero matches across all source files. ✅ clean. Damage exclusively flows through `UBBDamageExecution`.
7. **ASC on Character instead of PlayerState** — ASC is on `ABreachbornePlayerState`, character only delegates via `IAbilitySystemInterface::GetAbilitySystemComponent()`. ✅ clean.
8. **Skipping `OnRep_PlayerState`** — `AHunterCharacter::OnRep_PlayerState` AND `PossessedBy` both call `BindASCFromPlayerState()`. There's an additional retry timer on the client path for the case where PlayerState arrives late. ✅ matches Lyra's pattern.
9. **Spawning particles directly from ability code** — Ability code uses `BBGameplayTags::GameplayCue_Ghost_LMB_Impact` / `GameplayCue_Ghost_Q_Fire` instead of direct VFX spawning (cue tags are declared but the cue Blueprints themselves live in `Content/` and are unverifiable). ✅ pattern is correct in C++.
10. **Pre-optimizing with custom ReplicationGraph** — None present. ✅ clean.

**Other concerns I noticed worth flagging (not on the list):**
- `UFUNCTION(Server, Reliable)` is used everywhere; **none** use `WithValidation`. The networking cheatsheet recommends `WithValidation` for client-originated RPCs. CLAUDE.md doesn't make it mandatory and the project explicitly defers anti-cheat, but this is the norm spot-violation worth knowing.
- `ABreachbornePlayerController` mixes legitimate `Exec` debug commands with the production code path. They're guarded with `Server` RPCs that forward from the client, which is correct, but there's a *lot* of debug-only API here (>15 console commands) that will need to be `#if !UE_BUILD_SHIPPING` gated before any release.
- `BreachbornePlayerController.cpp::CreateConsumableInputActions` — programmatically creates `UInputAction`/`UInputMappingContext` UObjects at runtime instead of using data assets. Functional but unusual; see "files I'd want clarification on" below.

# Files I'd want clarification on

| File | Reason |
|---|---|
| `Source/Breachborne/Core/BreachbornePlayerState.cpp` | The proxy-struct attribute replication scheme (`Rep_HealthData`, etc.) is a substantial deliberate workaround for a UE subobject misrouting issue (documented in the comments on line 16-26 and 68-100). The `SpawnedAttributes` removal in `BBAbilitySystemComponent::GetLifetimeReplicatedProps` is the other half of the same workaround. This is non-standard GAS — most projects use `GAMEPLAYATTRIBUTE_REPNOTIFY` directly. Worth understanding whether this was the right long-term fix or a temporary patch. |
| `Source/Breachborne/Core/BreachbornePlayerController.cpp` (`CreateConsumableInputActions`) | Creates `UInputAction` and `UInputMappingContext` at runtime with `NewObject<>` because there are no `.uasset` files for them. This works but bypasses the standard data-driven flow and means consumable hotkeys are partly hardcoded. |
| `Source/Breachborne/Items/BBShopkeeper.cpp` | `ServerRequestPurchase_Implementation` builds an equipment definition via `NewObject<UBBEquipmentDefinition>()` on the fly with hardcoded stats, instead of looking up by `Listing.ItemID` from `UBBItemRegistry`. Inline comment says "(proper lookup via Registry in 5H)" — implies a planned refactor. |
| `Source/Breachborne/Core/BreachbornePlayerController.cpp` (`ServerDebugEquipTestItem_Implementation`) | Same pattern — production-looking code path but with placeholder stats. |
| `Source/Breachborne/Core/BreachborneGameMode.h` | `SquadSize = 3` despite both CLAUDE.md and phases.md specifying squad size 4. Is this a deliberate dev iteration setting or a bug? |
| `Source/Breachborne/Storm/BBStormManager.h` | Stores storm state as actor properties rather than as a `FStormCircleState` struct on `ABreachborneGameState`, as phases.md prescribes. Functionally OK but diverges from the architecture spec — was the spec changed? |
| `Source/Breachborne/Storm/StormShift_Default.cpp` | The "default" shift just shrinks toward map center — Phase 5 exit criteria require "one storm shift implemented and verifiably alters match conditions." The default doesn't *mutate* anything. Was this intended as a placeholder until Bullet Trains? |
| `Source/Breachborne/Combat/BBGlideBot.h/.cpp`, `BBAllyBot.h/.cpp`, `BBTargetDummy.h/.cpp` | These are test helpers but they're full ACharacter / AActor classes with their own ASC + AttributeSet. They live in `Combat/` next to production projectiles, not in a `Test/` or `Dev/` folder, and aren't gated with `#if WITH_EDITOR` or shipping macros. Will need cleanup before any release build. |
| `Source/Breachborne/Characters/BBCharacterMovementComponent.cpp` (`PhysGliding`) | The "sharp turn braking" logic (`>90° dot < 0`) reverses the velocity through zero before re-accelerating. This is an intentional design choice but not documented in `supervive-reference.md`'s gliding section. Worth verifying whether this matches the desired feel. |
| `Source/Breachborne/Abilities/BBAttributeSetBase.h/.cpp` | The `GetOwningAbilitySystemComponent()` shadowing technique is unusual and sits next to a tag-team workaround on PlayerState. Together they form a complex non-standard solution to GAS subobject replication. Documented in comments but worth understanding why the standard path didn't work. |

**Summary verdict:** Phases 0-2 are largely complete with minor gaps (squad size, prefix consistency); Phase 3 has the gliding/spike loop done but no real deathbox; Phase 4's wisp lifecycle is the strongest system but has nothing past wisp expiration; Phase 5 has structural plumbing but no real shift; Phase 6 has surprisingly mature inventory code despite Phase 4/5 being incomplete; Phases 7-10 are ❌ across the board with the exception of a debug HUD.
